    Control 1.22: Information Barriers for AI Agents

    Overview

    Control ID: 1.22 Control Name: Information Barriers for AI Agents Regulatory Reference: FINRA Rule 3110, SEC Regulation SHO, OCC 2011-12, FINRA Rule 5280 Setup Time: 3-5 hours

    Purpose

    Implement information barriers (also known as "Chinese walls" or "ethical walls") to prevent AI agents from crossing compliance boundaries between different business units. Investment banks, broker-dealers, and other financial institutions must maintain separation between functions such as research and trading, investment banking and sales, and public and private side activities. When AI agents have access to data across these boundaries, they can inadvertently enable information sharing that violates regulatory requirements.

    Critical for Investment Banks

    Organizations with investment banking, research, trading, and sales functions must implement information barriers. Failure to maintain separation can result in insider trading violations, front-running allegations, and regulatory sanctions.

    Description

    Information Barriers for AI Agents establishes controls to:

    • Prevent Cross-Barrier Data Access: Agents cannot access data from business units on opposite sides of a compliance wall
    • Enforce User Context Inheritance: Agents inherit the information barrier policies of their users
    • Audit Barrier Enforcement: Log and monitor all barrier-related access decisions
    • Support Wall-Crossing Procedures: Enable controlled, documented exceptions when required
    • Integrate with Microsoft Purview: Leverage Purview Information Barriers for consistent enforcement

    See Microsoft Purview Information Barriers for platform capabilities.

    Key Capabilities

    Capability Description FSI Relevance
    Barrier Policy Enforcement Prevent cross-barrier communication and data access Regulatory compliance
    User Context Inheritance Agents respect user's barrier assignments Consistent access control
    Barrier Crossing Audit Log all barrier-related decisions Investigation support
    Wall-Crossing Workflow Controlled exception process Business flexibility
    Cross-Platform Enforcement Barriers apply to SharePoint, Teams, and agents Comprehensive protection

    Governance Levels

    Baseline (Level 1)

    Requirement Configuration
    User segmentation Define user segments for barrier enforcement
    Basic barriers Block communication between defined segments
    Documentation Document barrier policies and segment assignments

    Minimum requirements:

    • Define user segments aligned with compliance walls
    • Configure basic Information Barrier policies
    • Document segment assignments and policy rationale
    • Manual review of agent access patterns quarterly

    Recommended (Level 2-3)

    Requirement Configuration
    Agent-aware barriers Extend barriers to Copilot and agent interactions
    SharePoint integration Barrier enforcement on knowledge sources
    Automated monitoring Alert on potential barrier violations
    Wall-crossing workflow Documented exception process

    FSI recommendations:

    • Configure Information Barriers for all Copilot interactions
    • Ensure SharePoint sites respect barrier policies
    • Implement automated alerting for barrier-related events
    • Establish formal wall-crossing approval workflow
    • Monthly review of barrier effectiveness

    Regulated/High-Risk (Level 4)

    Requirement Configuration
    Comprehensive barriers All data sources and agents enforce barriers
    Real-time monitoring Continuous barrier enforcement verification
    Wall-crossing audit Full audit trail of all exceptions
    Third-party validation Annual review of barrier effectiveness

    FSI considerations (high-risk):

    • Information Barriers mandatory for all Zone 3 agents
    • Real-time alerting on barrier events
    • Preserve barrier audit logs for 6+ years
    • Annual third-party assessment of barrier controls
    • Immediate escalation for barrier violations

    Setup & Configuration

    Step 1: Define User Segments

    Portal Path: Microsoft Purview Compliance Portal → Information Barriers → Segments

    1. Sign in to Microsoft Purview Compliance Portal
    2. Navigate to Information Barriers → Segments
    3. Click New segment
    4. Configure segment using Azure AD attributes:

    Example Segments for Investment Bank:

    Segment Name Definition Description
    IB-Research Department equals "Research" Equity research analysts
    IB-Trading Department equals "Trading" Trading desk personnel
    IB-InvestmentBanking Department equals "Investment Banking" Deal team members (private side)
    IB-Sales Department equals "Sales" Institutional sales
    IB-Compliance Department equals "Compliance" Compliance and legal

    PowerShell Segment Creation:

    # Connect to Information Protection
Connect-IPPSSession -UserPrincipalName admin@contoso.com

# Create segments
New-OrganizationSegment -Name "IB-Research" -UserGroupFilter "Department -eq 'Research'"
New-OrganizationSegment -Name "IB-Trading" -UserGroupFilter "Department -eq 'Trading'"
New-OrganizationSegment -Name "IB-InvestmentBanking" -UserGroupFilter "Department -eq 'Investment Banking'"
New-OrganizationSegment -Name "IB-Sales" -UserGroupFilter "Department -eq 'Sales'"
New-OrganizationSegment -Name "IB-Compliance" -UserGroupFilter "Department -eq 'Compliance'"

# Verify segments
Get-OrganizationSegment | Select-Object Name, UserGroupFilter

    Step 2: Configure Information Barrier Policies

    Portal Path: Microsoft Purview Compliance Portal → Information Barriers → Policies

    1. Navigate to Information Barriers → Policies
    2. Click Create policy
    3. Configure barrier rules:

    Example Barrier Policies:

    Policy Name Segment 1 Segment 2 Action
    Research-Trading Barrier IB-Research IB-Trading Block
    IB-Sales Barrier IB-InvestmentBanking IB-Sales Block
    Research-IB Barrier IB-Research IB-InvestmentBanking Block

    PowerShell Policy Creation:

    # Create barrier policies
New-InformationBarrierPolicy -Name "Research-Trading-Barrier" `
    -AssignedSegment "IB-Research" `
    -SegmentsBlocked "IB-Trading" `
    -State Active

New-InformationBarrierPolicy -Name "IB-Sales-Barrier" `
    -AssignedSegment "IB-InvestmentBanking" `
    -SegmentsBlocked "IB-Sales" `
    -State Active

New-InformationBarrierPolicy -Name "Research-IB-Barrier" `
    -AssignedSegment "IB-Research" `
    -SegmentsBlocked "IB-InvestmentBanking" `
    -State Active

# Apply policies
Start-InformationBarrierPoliciesApplication

# Check application status
Get-InformationBarrierPoliciesApplicationStatus

    Step 3: Configure Copilot and Agent Barrier Enforcement

    Ensuring Agents Respect Information Barriers:

    Microsoft 365 Copilot and Copilot Studio agents inherit the information barrier policies of the user invoking them. To ensure proper enforcement:

    1. Verify User Segment Assignment:
    2. Confirm all users are assigned to appropriate segments

    3. Users not in segments may bypass barriers

    4. Configure SharePoint Site Barriers:

    5. SharePoint sites used as agent knowledge sources must have barrier-compatible permissions

    6. Sites should be accessible only to appropriate segments

    7. Agent Knowledge Source Review:

    8. Audit agent knowledge sources for barrier compliance
    9. Ensure agents cannot access cross-barrier content

    Agent Barrier Audit Query:

    // Audit agent access across barrier boundaries
CopilotInteraction
| where TimeGenerated > ago(24h)
| extend UserSegment = tostring(parse_json(UserContext).Segment)
| extend DataSourceSegment = tostring(parse_json(DataContext).SourceSegment)
| where UserSegment != DataSourceSegment
| project TimeGenerated, UserId, UserSegment, AgentId, DataSourceSegment, Action
| summarize CrossBarrierAttempts = count() by UserSegment, DataSourceSegment, AgentId
| where CrossBarrierAttempts > 0
| order by CrossBarrierAttempts desc

    Step 4: Configure Wall-Crossing Procedures

    Wall-Crossing Workflow:

    When legitimate business needs require temporary barrier exceptions:

    Wall-Crossing Approval Workflow
================================

Trigger: Wall-crossing request submitted

Workflow:
├── Capture request details
│   ├── Requestor and segment
│   ├── Target segment/person
│   ├── Business justification
│   └── Duration requested
├── Route for approval
│   ├── Compliance Officer review (mandatory)
│   ├── Legal review (if material non-public information)
│   └── Business unit head (requesting side)
├── If approved:
│   ├── Document approval with timestamp
│   ├── Add user to restricted trading list (if applicable)
│   ├── Enable temporary access
│   ├── Set automatic expiration
│   └── Notify monitoring team
├── If rejected:
│   ├── Document rejection reason
│   └── Return to requestor
├── On expiration/completion:
│   ├── Remove wall-crossing access
│   ├── Verify restricted trading list status
│   └── Archive documentation
└── Retain records for 6+ years

    Wall-Crossing Documentation Template:

    Field Type Description
    Request ID Auto-number WC-YYYY-###
    Requestor Person Who is crossing the wall
    Source Segment Choice Their home segment
    Target Segment Choice Segment they need access to
    Business Justification Multi-line Why crossing is needed
    Start Date Date/Time When crossing begins
    End Date Date/Time When crossing ends
    Approved By Person Compliance approver
    Restrictions Multi-line Trading restrictions applied
    Status Choice Active, Completed, Expired

    Step 5: Configure Barrier Monitoring and Alerting

    Barrier Violation Detection:

    // Detect potential barrier violations in agent interactions
let barrier_pairs = dynamic([
    {"source": "Research", "target": "Trading"},
    {"source": "InvestmentBanking", "target": "Sales"},
    {"source": "Research", "target": "InvestmentBanking"}
]);

CopilotInteraction
| where TimeGenerated > ago(1h)
| extend UserDepartment = tostring(parse_json(UserContext).Department)
| extend ContentSource = tostring(parse_json(DataContext).SourceDepartment)
| where UserDepartment != ContentSource
| extend IsCrossBarrier = iff(
    (UserDepartment == "Research" and ContentSource == "Trading") or
    (UserDepartment == "Trading" and ContentSource == "Research") or
    (UserDepartment == "InvestmentBanking" and ContentSource == "Sales") or
    (UserDepartment == "Sales" and ContentSource == "InvestmentBanking"),
    true, false)
| where IsCrossBarrier == true
| project TimeGenerated, UserId, UserDepartment, AgentId, ContentSource, UserInput
| order by TimeGenerated desc

    Alert Configuration:

    Alert Condition Action
    Barrier Violation Attempt Cross-barrier data access detected Alert Compliance immediately
    Unusual Cross-Segment Query Agent query references multiple segments Review within 24h
    Wall-Crossing Expiration Active wall-crossing approaching end Notify requestor 24h before
    Segment Assignment Gap User without segment assignment Alert IT for remediation

    Step 6: Zone-Specific Configuration

    Zone 1 (Personal Productivity):

    Configuration:
- Information Barriers: Inherit user's segment
- Agent Access: Personal data only
- Monitoring: Standard logging
Rationale: Personal agents have limited scope; user barriers apply automatically

    Zone 2 (Team Collaboration):

    Configuration:
- Information Barriers: Enforce team segment boundaries
- Agent Access: Team SharePoint only (barrier-compliant)
- Monitoring: Weekly barrier compliance review
- Alert: On cross-team access attempts
Rationale: Team agents may access shared resources; barrier enforcement critical

    Zone 3 (Enterprise Managed):

    Configuration:
- Information Barriers: MANDATORY enforcement
- Agent Access: Barrier-validated sources only
- Monitoring: Real-time barrier enforcement
- Alert: Immediate on any barrier event
- Wall-Crossing: Formal approval required
- Audit: Full evidence chain preserved
Rationale: Maximum protection for customer-facing and regulated agents

    PowerShell Configuration

    Comprehensive Barrier Audit Script

    # ============================================================
# Control 1.22: Information Barriers for AI Agents
# Barrier Compliance Audit Script
# ============================================================

param(
    [string]$OutputPath = ".\IB-Audit-$(Get-Date -Format 'yyyyMMdd')"
)

New-Item -ItemType Directory -Path $OutputPath -Force | Out-Null

Write-Host "=== Information Barriers Audit ===" -ForegroundColor Cyan

# ============================================================
# SECTION 1: Segment Inventory
# ============================================================

Write-Host "`nSection 1: Retrieving segments..." -ForegroundColor Yellow

Connect-IPPSSession -UserPrincipalName admin@contoso.com

$segments = Get-OrganizationSegment
$segments | Select-Object Name, UserGroupFilter, CreatedDate |
    Export-Csv "$OutputPath\Segments.csv" -NoTypeInformation

Write-Host "Found $($segments.Count) segments" -ForegroundColor Green

# ============================================================
# SECTION 2: Policy Inventory
# ============================================================

Write-Host "`nSection 2: Retrieving policies..." -ForegroundColor Yellow

$policies = Get-InformationBarrierPolicy
$policies | Select-Object Name, AssignedSegment, SegmentsBlocked, State |
    Export-Csv "$OutputPath\Policies.csv" -NoTypeInformation

Write-Host "Found $($policies.Count) policies" -ForegroundColor Green

# ============================================================
# SECTION 3: User Segment Coverage
# ============================================================

Write-Host "`nSection 3: Checking user segment coverage..." -ForegroundColor Yellow

$allUsers = Get-User -ResultSize Unlimited
$usersWithSegment = 0
$usersWithoutSegment = @()

foreach ($user in $allUsers) {
    $recipientInfo = Get-Recipient $user.UserPrincipalName -ErrorAction SilentlyContinue
    if ($recipientInfo.InformationBarrierSegment) {
        $usersWithSegment++
    } else {
        $usersWithoutSegment += $user.UserPrincipalName
    }
}

Write-Host "Users with segment: $usersWithSegment" -ForegroundColor Green
Write-Host "Users without segment: $($usersWithoutSegment.Count)" -ForegroundColor $(if ($usersWithoutSegment.Count -gt 0) { "Yellow" } else { "Green" })

if ($usersWithoutSegment.Count -gt 0) {
    $usersWithoutSegment | Out-File "$OutputPath\Users-Without-Segment.txt"
    Write-Host "WARNING: Users without segments may bypass barriers" -ForegroundColor Yellow
}

# ============================================================
# SECTION 4: Policy Application Status
# ============================================================

Write-Host "`nSection 4: Checking policy application status..." -ForegroundColor Yellow

$appStatus = Get-InformationBarrierPoliciesApplicationStatus
Write-Host "Last application: $($appStatus.ApplicationDate)" -ForegroundColor Cyan
Write-Host "Status: $($appStatus.Status)" -ForegroundColor $(if ($appStatus.Status -eq "Completed") { "Green" } else { "Yellow" })

# ============================================================
# SECTION 5: Generate Compliance Report
# ============================================================

Write-Host "`nSection 5: Generating compliance report..." -ForegroundColor Yellow

$html = @"
<!DOCTYPE html>
<html>
<head>
    <title>Information Barriers Audit Report</title>
    <style>
        body { font-family: 'Segoe UI', sans-serif; margin: 40px; }

        h2 { color: #333; border-bottom: 2px solid #0078d4; padding-bottom: 5px; }
        .dashboard { display: flex; gap: 20px; flex-wrap: wrap; margin: 20px 0; }
        .card { padding: 20px; background: #f5f5f5; border-radius: 8px; min-width: 150px; }
        .card.warning { background: #fff3cd; border-left: 4px solid #ffc107; }
        .card.success { background: #d4edda; border-left: 4px solid #28a745; }

        th, td { padding: 10px; text-align: left; border-bottom: 1px solid #ddd; }
        th { background: #0078d4; color: white; }
    </style>
</head>
<body>
    <h1>Information Barriers Audit Report</h1>
    <p>Control 1.22 | Generated: $(Get-Date -Format "yyyy-MM-dd HH:mm:ss")</p>

    <h2>Summary</h2>
    <div class="dashboard">
        <div class="card"><h3>Segments</h3><p style="font-size:28px;">$($segments.Count)</p></div>

    Troubleshooting & Validation

    Issue: Users Not Assigned to Segments

    Symptoms: Users can access cross-barrier content

    Resolution:

    1. Identify users without segment assignment (run audit script)
    2. Update Azure AD attributes to match segment filters
    3. Re-run segment processing: Start-InformationBarrierPoliciesApplication
    4. Verify user segment assignment: Get-Recipient user@contoso.com | Select InformationBarrierSegment
    5. Document gap and remediation for compliance

    Issue: Agent Accessing Cross-Barrier Content

    Symptoms: Agent returns data from blocked segments

    Resolution:

    1. Verify user invoking agent has correct segment assignment
    2. Check agent knowledge source permissions
    3. Confirm SharePoint sites have barrier-compatible permissions
    4. Review agent data connections for cross-barrier sources
    5. Remove non-compliant knowledge sources from agent

    Issue: Barrier Policies Not Applying

    Symptoms: Policies show "Active" but not enforced

    Resolution:

    1. Check policy application status: Get-InformationBarrierPoliciesApplicationStatus
    2. If not "Completed", re-run: Start-InformationBarrierPoliciesApplication
    3. Wait for processing (can take several hours)
    4. Verify segment membership is correctly calculated
    5. Escalate to Microsoft support if issue persists

    Issue: Wall-Crossing Access Not Expiring

    Symptoms: Temporary access continues after approved period

    Resolution:

    1. Verify wall-crossing end date in workflow system
    2. Check automated expiration job is running
    3. Manually remove access if expired
    4. Update restricted trading list status
    5. Document incident and implement process improvement
